# v9.0 — Paper trading data-integrity audit

A summarized record of what this release was asked to deliver and what actually shipped.

## Trigger

The user reported that the Paper Trading page could show a fully-reset account ($10,000
balance, $0 P&L, 0 open, 0 closed, empty Auto Trade Log) while the Strategy Journal still showed
completed JVM trades — an apparent inconsistency between two pages that should agree.

## Key requirements

- **Investigate before modifying anything.** Trace the actual relationship between
  `paperAccount`, `journalEntries`, `autoTrading.log`, `save()`/`loadSaved()`, the existing
  `resetPaperAccount()`, and every journaling hook, before writing any fix.
- **Do not fabricate missing paper positions. Do not silently rebuild `paperAccount` from
  `journalEntries`. Do not delete journal history automatically.**
- Add a safe, **display-only** classification layer so every JVM journal record shows its real
  relationship to the account: active position, closed account trade, journal only, account
  reset history, manual entry, legacy/imported, or developer test.
- Update the Paper Trading mini-journal to show each row's source/relationship/trade ID.
- Replace the single ambiguous "Reset Account" confirmation with a clear 3-option modal (reset
  active account only / full reset including linked journal records / cancel), explaining
  exactly what is deleted vs. preserved in each case.
- Add an account-reset-history marker so preserved-but-orphaned entries read as "Account reset
  history" instead of an unexplained disconnect.
- **Do not change**: trading methodology, entry/stop/target logic, sizing, strategy identifiers,
  ALEX state, Academy state, chart/drawing state.
- Create a durable regression-baseline artifact **inside the project** (not a scratchpad temp
  file) capturing protected trading-function source hashes, known-good-file hash, and fixture
  counts — because two prior ad-hoc baseline files had already been found missing between
  sessions.
- Full syntax validation, all pre-existing fixtures, new fixtures for the classification/reset
  behavior, version bump, and an exact changelog entry.

## What was delivered

- **Root cause found**: `paperAccount` and `journalEntries` were always two fully independent
  stores; `resetPaperAccount()` only ever wiped the former, by design — but the old single
  `confirm()` dialog never explained that, and orphaned journal rows had no way to show why they
  no longer matched the account.
- `classifyJvmJournalRecord(r)` — the pure, read-only classification function described above.
  One real bug was caught and fixed before shipping: the first draft checked `r.tradeId==null`
  to detect a manual entry, but the normalizer always backfills `tradeId` from the raw record's
  own `id` for manual rows, so that check could never fire — fixed to use the already-computed
  `isLegacyManual` flag instead.
- `renderPaperMiniJournal()` — the updated Paper Trading mini-journal, deliberately built
  separately from the shared journal renderer so the unified Journal tab and ALEX's own
  mini-journal stayed untouched.
- The 3-option reset modal (`openPaperResetModal()`, `confirmPaperResetAccountOnly()`,
  `confirmPaperResetFull()`), each recording a `paperResetHistory` entry before saving.
- [`regression-baseline-tools.py`](../../regression-baseline-tools.py) and
  [`regression-baseline.json`](../../regression-baseline.json), committed inside the repository —
  see [TESTING.md](../TESTING.md).

## Verification

20 new fixtures (all 7 classification categories, both reset options' exact effects proven
independently, reset-history entries recorded correctly, backward compatibility with a record
that never had a `tradeId`, and — critically — that neither reset function nor the classifier
itself ever mutates ALEX or Academy state), plus the complete pre-existing 293-fixture suite,
unchanged (313 total). Known-good hash unchanged.
