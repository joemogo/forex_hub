# v11.0.1 — Paper ledger transaction correction

A summarized record of what this release was asked to deliver and what actually shipped. See
[INCIDENTS.md](../INCIDENTS.md#inc-001) for the incident writeup and
[ADR-003](../adr/ADR-003-paper-ledger-transaction-model.md) for the design decision this release
established.

## Trigger

An independent review of the actual v11.0 code (not a new user-reported symptom) found a more
precise defect than v11.0's own fix addressed.

## Key requirements

- **Fix this as a ledger-transaction problem, not a classification problem.**
- Root cause identified by the reviewer: `save()` wrote `fxhub_journal` unconditionally before
  attempting the guarded `paperAccount` write, which could still legitimately be rejected — a
  split transaction that could still leave a `JOURNAL_ONLY` orphan even with v11.0's version
  guard in place. Separately, the guard ran on every general `save()` call, so unrelated activity
  in one tab could falsely stale-out a different, actively-trading tab.
- **General `save()` must not write or version `paperAccount` unless the paper account actually
  changed.**
- Create a **dedicated paper-ledger commit function**, used only by actual paper-account
  mutations: `openPaperPosition`, `closePaperPosition`, `setPaperBalance`, reset actions,
  test-trade clearing, reconciliation.
- The paper-account mutation and its linked JVM journal update must commit atomically from the
  application's perspective; **if the account commit is rejected**: do not persist the linked
  journal mutation, do not mark `tradedToday`, do not append the auto-trade log, do not show
  "trade placed," roll back the in-memory account and journal mutations, and surface a visible
  blocking error telling the user to reload.
- `openPaperPosition` must return an explicit success/failure result confirming actual
  persistence, not just successful in-memory construction.
- `checkAutoTrades` must only update `tradedToday`/log/unseenCount/notifications after the
  paper-ledger transaction succeeds.
- `closePaperPosition` must only leave the journal record closed if the closed position, balance,
  and journal closure were all successfully committed.
- **Do not increment `fxhub_paper_version` for unrelated scanner/alert/checklist/Academy/chart/
  general-journal saves.**
- Preserve existing journal-only records — do not automatically restore them; require verified
  fields and explicit user confirmation.
- Fixtures proving 10 specific properties, including a two-tab conflict producing a *visible
  failure*, never a false "successful trade."
- **"Do not continue feature development until this is fixed and reproduced through the real
  browser lifecycle."**

## What was delivered

- `save()` no longer touches `paperAccount`/`fxhub_paper`/`fxhub_paper_version` at all.
- `commitPaperLedger()` — the sole paper-ledger persistence path: writes the guarded
  paper-account state first, then persists everything else the transaction touched only if that
  succeeds.
- Every listed mutation call site (including `generateTestPaperTrade`'s post-open TEST tagging,
  and the Diagnostics self-test's own isolated-account cleanup path, both found during
  implementation to need the same treatment) now snapshots, mutates, commits, and rolls back on
  rejection.
- `openPaperPosition` returns `{committed:true, ...}` on success or `{error, blocked:true}` on
  rejection; `closePaperPosition` returns `{closedPos, committed:true}` or `{error,
  blocked:true}` the same way.
- A new, always-visible (not Developer-Mode-gated) red blocking banner on Paper Trading
  (`paperLedgerBlockingError` / `renderPaperLedgerBlockingBanner()`).

## Verification

15 new fixtures (two pre-existing v11.0 fixtures updated in place to call the new
`commitPaperLedger()`, since the bare-`save()` path they exercised no longer touches
`paperAccount` by design), plus the complete pre-existing 352-fixture suite, unchanged (367
total). Live browser verification of: a blocked `closePaperPosition()` rolling back both
`paperAccount` and `journalEntries`; a real `checkAutoTrades()` call under a rigged version
conflict leaving `tradedToday`/log/account/journal completely untouched; the full 3-trade clean
lifecycle still passing; unrelated `save()` calls confirmed not to advance the paper version; a
reload retaining all committed data; and the blocking banner rendering correctly (screenshotted)
with the balance left untouched. Baseline comparison disclosed exactly `openPaperPosition`/
`closePaperPosition` as changed, math confirmed byte-identical. Known-good hash unchanged.
