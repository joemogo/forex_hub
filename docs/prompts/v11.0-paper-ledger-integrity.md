# v11.0 — Paper Ledger Integrity (urgent data-integrity investigation)

A summarized record of what this release was asked to deliver and what actually shipped. See
[INCIDENTS.md](../INCIDENTS.md#inc-001) for the incident writeup this release is part of.

## Trigger

The user reported three EUR/USD trades, placed and completed *after* a paper account reset, that
showed real `Win`/`+2.00R` results in the journal but classified `JOURNAL_ONLY` — while Paper
Trading showed a fully-reset $10,000/0-open/0-closed account. This proved new trade events were
reaching `journalEntries` but the linked `paperAccount` record was being lost — a live, active
defect, not historical/pre-v9.0 data.

## Key requirements (urgent, explicit)

- **Do not reclassify these records as Account Reset History. Do not hide or delete them. Do not
  reconstruct positions from journal data until the root cause is proven.**
- A forensic trace of the actual three records: every stored field, whether each journaling hook
  ran, whether a matching position ever existed in the account.
- Trace the complete live open→close lifecycle (14 numbered steps from `checkAutoTrades()`
  through the final `save()`/render).
- Explicitly test 13 named candidate failure modes (a race between open/journal/save ordering, a
  stale in-memory object overwriting a newer save, `loadSaved()` running twice, the reset modal
  running without user action, a swallowed exception, a duplicate-close race, etc.) — with the
  explicit warning that the protected-function baseline being unchanged does **not** by itself
  prove the surrounding save/load/reset code is safe.
- Temporary, developer-only lifecycle logging (10 checkpoints) — explicitly: **"Do not expose API
  keys or credentials."**
- Reproduce one full synthetic trade through the real open/close engine using that
  instrumentation.
- A fix ensuring: every opened trade exists in `openPositions`; every completed trade transfers
  atomically to `closedPositions`; balance/P&L update exactly once; the account is persisted only
  after all mutations finish; a later stale save cannot overwrite newer state; duplicate close
  calls can't double-process; errors are surfaced in Diagnostics rather than swallowed — with the
  explicit constraint: **"Do not silently rebuild the current paper account from journal records
  as the primary fix."**
- A separate, explicit-confirmation-gated reconciliation preview/apply tool for the 3 affected
  records, with its own audit trail.
- A new "Paper Ledger Integrity" Diagnostics view (orphans in both directions, duplicate IDs,
  missing P&L, a running balance reconciliation), with a new post-reset `JOURNAL_ONLY` record
  required to trigger a visible red warning.
- Fixtures proving 13 specific properties of the fixed lifecycle.
- **"Do not bump the version or declare completion until the actual root cause is identified and
  reproduced."**

## What was delivered

- **Root cause, reproduced live**: `save()` wrote `paperAccount` with no staleness/version check
  at all. Reproduced by opening+closing a real trade cleanly, then reassigning the in-memory
  `paperAccount` to a stale snapshot and calling `save()` — which silently overwrote the correct,
  newer data.
- A monotonic version guard (`fxhub_paper_version` + `savePaperAccountGuarded()`), an in-flight
  duplicate-close guard (`paperPositionsClosing`), developer-mode-gated lifecycle logging
  (`logPaperLifecycle()` — a true no-op unless Developer Mode is on), and
  `recordPaperEngineError()` replacing a previously-silent `catch(e){}` in `checkAutoTrades()`.
- `computePaperLedgerIntegrity()`, the reconciliation preview/apply flow
  (`computeReconciliationPreview()`/`applyPaperReconciliation()`), and the Paper Ledger Integrity
  Diagnostics card, including its red warning banner for genuinely orphaned trades.

## What v11.0 did *not* fully resolve

An independent review after this release found a more precise mechanism — a split transaction
between the journal write and the guarded account write — that this release's version guard alone
did not close. See [v11.0.1's own record](v11.0.1-ledger-transaction-correction.md) and
[INC-001](../INCIDENTS.md#inc-001) for the full correction.

## Verification

17 new fixtures, plus the complete pre-existing 335-fixture suite, unchanged (352 total). Live
browser reproduction of both the original bug and the fix blocking it; a full open→close
lifecycle for three sequential trades; sequential and concurrent duplicate-close idempotency; the
Diagnostics card's red-banner and reconciliation flow end-to-end. Known-good hash unchanged.
