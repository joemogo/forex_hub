# MOGO

MOGO is a single-file, browser-based forex trading research application. It connects to an
OANDA account (practice or live) for market data, runs two independently-tracked trading
strategies against that data in a simulated ("paper") account, and gives you tools to journal,
review, and study those trades.

Everything — markup, styles, and application logic — lives in one file: [`index.html`](index.html).
There is no build step and no backend; open the file (or serve it statically) in a browser and it
runs entirely client-side, persisting its state in the browser's `localStorage`.

## Current version

The running application version is defined in-code and shown on the Diagnostics page:

```js
const APP_VERSION='11.0.1';
```

The full, unabridged history of every release is preserved in-code as `APP_VERSION_LOG` near the
top of `index.html`. [docs/RELEASE_NOTES.md](docs/RELEASE_NOTES.md) is a readable, maintained
summary of that same history — see that file if you want the short version, or the in-code log if
you want the original, complete, as-shipped changelog entry for a specific release.

## Two strategies, one app

MOGO runs two trading strategies side by side, each fully isolated from the other in code and in
storage:

- **JVM** — the app's original ("current") strategy, internally identified as `current_strategy`
  / `JVM` in stored records and code.
- **ALEX** — a second, independently-specified strategy, internally identified as
  `alex_g_sr_v1` / `ALEX`.

**"MOGO" is the product name. "JVM" and "ALEX" are internal strategy identifiers, not product
names.** See [docs/PRODUCT_VISION.md](docs/PRODUCT_VISION.md) and
[docs/adr/ADR-001-product-name-vs-strategy-identifier.md](docs/adr/ADR-001-product-name-vs-strategy-identifier.md)
for why that distinction is deliberate and preserved.

Both strategies' entry/stop/target/direction/qualification/exit logic is treated as frozen
methodology — see [docs/CODING_STANDARDS.md](docs/CODING_STANDARDS.md).

## Running it

There is no build or install step.

- **Directly**: open `index.html` in a browser. Some browsers restrict `localStorage`/`fetch`
  behavior for `file://` pages — if data isn't persisting, serve it instead.
- **Served locally**: from this directory, `python3 -m http.server 8940` and open
  `http://localhost:8940/index.html`.

On first load you'll see a connect screen asking for an OANDA API key, account ID, and
environment (practice or live). See [docs/STORAGE_KEYS.md](docs/STORAGE_KEYS.md) for exactly what
is and isn't persisted, and note that your OANDA API key is **never** written to `localStorage` —
it lives only in memory for the current browser session.

## Rule for future releases

Every release that changes behavior must update `docs/RELEASE_NOTES.md`, `docs/TESTING.md`, and
`docs/KNOWN_ISSUES.md`, plus any ADR or architecture document its change actually affects. See
the "Documentation requirements" note at the bottom of
[docs/RELEASE_NOTES.md](docs/RELEASE_NOTES.md) for the exact rule.

## Regression baseline

This project keeps a durable, committed regression baseline ([`regression-baseline.json`](regression-baseline.json),
generated by [`regression-baseline-tools.py`](regression-baseline-tools.py)) that hashes every
function and constant considered frozen trading methodology, so an accidental change to either
strategy's logic is caught automatically rather than relying on memory or review alone. See
[docs/TESTING.md](docs/TESTING.md) for how to run it.
